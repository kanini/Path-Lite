# Bug Fixing Agent Orchestrator

**Target Persona**: Software Developers, Full Stack Engineers, QA Engineers

Orchestrates the bug resolution phase: bug triage and task generation, test impact analysis, fix implementation with analysis and reviews, conditional test updates, and full test suite validation. Consumes bug reports from tracking systems or direct input.

**Prerequisite**: Bug report, issue URL, error log, or bug description.

**Handoff**: After successful test suite run and code review completion, address any findings before PR/deployment.

## Input Parameter: $ARGUMENTS (Mandatory)
**Accepts:** Bug report file | Bug URL | Issue description | Error log | Bug ID

**Supported Patterns:**
- Bug ID: `bug_XXX` | `BUG-XXX` (3-4 digits)
- Issue ID: `issue_XXX` | `ISSUE-XXX` (3-5 digits)
- GitHub/GitLab: `#XXX`
- JIRA format: `PROJECT-XXX`
- File path: `.propel/context/tasks/bug_XXX/task_*.md`

**Optional Parameters:**
- `--skip-history`: Skip git history analysis and findings registry lookup for faster execution (default: false)

### Input Processing Instructions
**CRITICAL**: Before proceeding with orchestration, you MUST determine input type and process accordingly:

1. **Bug ID Detection**: Check if `$ARGUMENTS` matches:
   - Pattern: `bug_\d{3,4}` or `BUG[-_]?\d{3,4}`
   - Pattern: `issue_\d{3,5}` or `ISSUE[-_]?\d{3,5}`
   - Pattern: `#\d+` (GitHub/GitLab style)
   - Pattern: `[A-Z]+-\d+` (JIRA format)
   - File path containing `bug_` or `issue_`

## Workflow Steps

### Bug Fix Path
1. **Plan Bug Resolution** - Triage bug and generate fix task
2. **Identify Impacted Unit Tests** - Analyze test impact from affected files
3. **Implement Bug Fix** - Execute the fix task
   - 3.1. **Analyze Implementation** - Review implementation against task requirements
   - 3.2. **Analyze UX** (CONDITIONAL) - Evaluate UI/UX if task has UI impact
   - 3.3. **Review Code** - Comprehensive code review
4. **Update Existing Unit Tests** (CONDITIONAL) - Update tests if impacted
   - 4.1. **Analyze Test Implementation** - Review test updates
5. **Run Full Test Suite** - Validate no regressions introduced

## Output
- Artifacts generated by each sub-workflow:
  - Step 1: `.propel/context/tasks/bug_<ID>/task_*.md`
  - Step 3.1: `.propel/context/tasks/bug_<ID>/reviews/task-review-*.md`
  - Step 3.2 (if UI): `.propel/context/tasks/bug_<ID>/reviews/ui-review-*.md`
  - Step 3.3: `.propel/code-reviews/review_*.md`
  - Step 4.1 (if tests updated): `.propel/context/tasks/bug_<ID>/reviews/task-review-test_*.md`
- Print progress after each step (e.g., "Step 1/5 completed", "Step 3.1 completed")
- Print key findings summary for each completed step
- Print final recommendation to review PR and merge

## Orchestration Instructions

When invoked, execute the following workflow:

---

### Step 1: Plan Bug Resolution
- **Invoke**: `.windsurf/workflows/plan-bug-resolution.md` with `$ARGUMENTS`
- **Input**: Bug report, URL, issue description, or error log
- **Wait**: Allow workflow to complete fully
- **Capture**: Generated task file path for subsequent steps
- **Summarize**: Present key findings including:
  - Bug ID extracted or assigned
  - Root cause analysis summary
  - Impact assessment (Critical/High/Medium/Low)
  - Affected files and components
  - Fix strategy overview
  - Regression risk assessment
- **Progress**: Print "Step 1/5 completed: Plan Bug Resolution"
- **Next**: Proceed to Test Impact Analysis

---

### Step 2: Identify Impacted Unit Tests
- **Purpose**: Analyze which existing unit tests may need updates based on the fix
- **Execution**:
  1. Read the generated task file from Step 1
  2. Extract list of files to be modified from the fix task
  3. For each affected file, search for existing unit tests:
     ```
     Grep("import.*<affected_file>|from.*<affected_file>", "**/*.test.*", "**/*.spec.*", "**/*Test.*")
     ```
  4. Identify test files that directly test the affected components
  5. Analyze if the fix changes behavior that tests verify
- **Determine TEST_UPDATE_REQUIRED**:
  - Set `TEST_UPDATE_REQUIRED = true` if:
    - Existing tests cover the bug's code path
    - Fix changes expected behavior (not just fixing incorrect behavior)
    - Test assertions may need updating due to fix
  - Set `TEST_UPDATE_REQUIRED = false` if:
    - No existing tests cover affected code
    - Fix corrects behavior to match existing test expectations
    - Tests are already failing due to the bug (fix will make them pass)
- **Summarize**: Present key findings including:
  - Number of test files identified
  - Test files directly impacted: [list]
  - TEST_UPDATE_REQUIRED: [true/false]
  - Rationale for test update decision
- **Progress**: Print "Step 2/5 completed: Identify Impacted Unit Tests"
- **Next**: Proceed to Bug Fix Implementation

---

### Step 3: Implement Bug Fix Task
- **Invoke**: `.windsurf/workflows/implement-tasks.md` with task file from Step 1
- **Input**: Task file from `.propel/context/tasks/bug_<ID>/task_*.md`
- **Wait**: Allow workflow to complete fully
- **Summarize**: Present key findings including:
  - Files created/modified
  - Requirements implemented
  - Build and test status
  - Todo items completed
- **Progress**: Print "Step 3/5 - Step 3.0 completed: Implement Bug Fix"
- **Next**: Proceed to Analyze Implementation

#### Step 3.1: Analyze Implementation
- **Invoke**: `.windsurf/workflows/analyze-implementation.md` with task file from Step 1
- **Input**: Same task file used in Step 3
- **Wait**: Allow workflow to complete fully
- **Purpose**: Verifies that the **TODOs defined in the task file** were implemented correctly
  - Each task file contains specific TODOs derived from root cause analysis
  - This step checks task-level completion and fix correctness
- **Summarize**: Present key findings including:
  - TODO completion status (completed vs pending)
  - Requirements fulfillment percentage (task-level)
  - Gap analysis (missing implementations)
  - Code quality assessment
  - Recommended actions for any gaps
- **Progress**: Print "Step 3/5 - Step 3.1 completed: Analyze Implementation"

#### Step 3.2: Analyze UX (CONDITIONAL)
- **UI Impact Gate**: Check task file's "Design References" table for `UI Impact` value:
  - If `UI Impact = Yes`: Proceed with UX analysis
  - If `UI Impact = No` or table missing: Skip UX analysis
- **IF UI impact detected**:
  - **Invoke**: `.windsurf/workflows/analyze-ux.md` with parameters:
    - `task_file`: Task file path (e.g., `.propel/context/tasks/bug_XXX/task_001_fix_*.md`)
      - Workflow will automatically extract: Figma URL, Hi-Fi Wireframe path, Screen ID, UXR Requirements
    - `server_url`: Development server URL (default: http://localhost:3000)
  - **Wait**: Allow workflow to complete fully
  - **Summarize**: Present key findings including:
    - Visual consistency score (design token deviations)
    - Hi-Fi wireframe comparison results (pixel deviations documented)
    - State coverage score ([N]/5 states implemented)
    - Accessibility compliance (WCAG 2.1 AA)
    - Responsive design validation (desktop/tablet/mobile)
    - Interaction quality assessment (hover/focus/active states)
    - UXR requirements validation checklist
  - **Progress**: Print "Step 3/5 - Step 3.2 completed: Analyze UX"
- **IF NO UI impact detected**:
  - Print "No UI impact detected in task (UI Impact = No). Skipping UX analysis."
  - **Progress**: Print "Step 3/5 - Step 3.2 skipped: No UI impact"

#### Step 3.3: Review Code
- **Invoke**: `.windsurf/workflows/review-code.md`
- **Input**: Reviews local changes (files modified during implementation)
- **Wait**: Allow workflow to complete fully
- **Summarize**: Present key findings including:
  - Files reviewed count
  - Findings by severity (Critical/High/Medium/Low)
  - Security coverage (OWASP checks)
  - Architecture assessment
  - Regression risk evaluation
  - Recommendation (Approve/Approve with Changes/Reject)
- **Progress**: Print "Step 3/5 - Step 3.3 completed: Review Code"
- **Gate**: If Critical findings exist, pause and ask user how to proceed

---

### Step 4: Update Existing Unit Tests (CONDITIONAL)

**Test Update Gate**: Check `TEST_UPDATE_REQUIRED` flag from Step 2:
- If `TEST_UPDATE_REQUIRED = true`: Proceed with test updates
- If `TEST_UPDATE_REQUIRED = false`: Skip to Step 5

- **IF test updates required**:
  - **Create Test Update Context**: Generate a test update task based on Step 2 analysis:
    - List test files requiring updates
    - Describe behavior changes from the fix
    - Specify assertion updates needed
  - **Invoke**: `.windsurf/workflows/implement-tasks.md` with test update context
  - **Wait**: Allow workflow to complete fully
  - **Summarize**: Present key findings including:
    - Test files updated
    - Assertions modified
    - New test cases added (if any)
    - All tests passing status
  - **Progress**: Print "Step 4/5 - Step 4.0 completed: Update Existing Unit Tests"

  #### Step 4.1: Analyze Test Implementation
  - **Invoke**: `.windsurf/workflows/analyze-implementation.md` with test update context
  - **Wait**: Allow workflow to complete fully
  - **Summarize**: Present key findings including:
    - Todo completion status (all test updates implemented)
    - Test modifications vs planned
    - Gaps identified (missing updates or coverage)
    - Recommended actions for any gaps
  - **Progress**: Print "Step 4/5 - Step 4.1 completed: Analyze Test Implementation"

- **IF NO test updates required**:
  - Print "No test updates required. Existing tests cover the fix correctly."
  - **Progress**: Print "Step 4/5 skipped: No test updates needed"

---

### Step 5: Run Full Test Suite
- **Purpose**: Validate that the bug fix introduces no regressions
- **Execution**:
  1. Detect project test framework:
     - Check for `package.json` → `npm test` or `npm run test`
     - Check for `*.csproj` → `dotnet test`
     - Check for `pytest.ini` or `setup.py` → `pytest`
     - Check for `pom.xml` → `mvn test`
     - Check for `build.gradle` → `gradle test`
  2. Execute full test suite with coverage reporting if available
  3. Capture test results and any failures
- **Wait**: Allow test suite to complete fully
- **Summarize**: Present key findings including:
  - Total tests run
  - Tests passed / failed / skipped
  - Code coverage percentage (if available)
  - Any test failures with details
  - Regression detection status
- **Gate**: All tests MUST pass
  - If failures exist: Pause and ask user how to proceed
  - Options: Fix failing tests, investigate regression, or abort
- **Progress**: Print "Step 5/5 completed: Run Full Test Suite"

---

### Final Summary
After all steps complete:

- Present overall bug fix completion summary
- Report workflow path executed (with conditional steps noted)
- List all artifacts generated with file paths:
  - Bug fix task: `.propel/context/tasks/bug_<ID>/task_*.md`
  - Implementation reviews: `.propel/context/tasks/bug_<ID>/reviews/task-review-*.md`
  - UI reviews (if applicable): `.propel/context/tasks/bug_<ID>/reviews/ui-review-*.md`
  - Code reviews: `.propel/code-reviews/review_*.md`
  - Test implementation reviews (if applicable): `.propel/context/tasks/bug_<ID>/reviews/task-review-test_*.md`
- Report test suite results:
  - Total tests: [N]
  - Passed: [N] | Failed: [N] | Skipped: [N]
  - Coverage: [X]% (if available)
- Highlight any [UNCLEAR] items or failed gates requiring attention
- Provide recommendation: "Review code review findings and merge PR after approval."

## Execution Rules

1. **Sequential Execution**: Complete each step and sub-step fully before proceeding to the next
2. **Conditional UX Analysis**: Step 3.2 (Analyze UX) executes only if UI impact detected in task
3. **Conditional Test Updates**: Step 4 executes only if TEST_UPDATE_REQUIRED = true from Step 2
4. **State Preservation**: All results are saved to `.propel/context/` directories
5. **Explicit Context**: Each workflow references previous step outputs explicitly
6. **Error Handling**: If a step fails, stop execution and ask user how to proceed
7. **Critical Gate**: If code review finds Critical severity issues, pause for user decision
8. **Test Suite Gate**: If any tests fail in Step 5, pause for user decision
9. **No Quality Gates at Orchestrator Level**: Quality evaluation is handled by individual sub-workflows
