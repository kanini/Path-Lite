# User Story - US_049

## Story ID
* ID: US_049

## Story Title
* Modular Form Schema Architecture

## Description
* As a developer, I want a modular form schema architecture with plugin-based design, so that additional treatment types can be added in the future without major refactoring.

## Acceptance Criteria
1. **Given** I need extensibility, **When** I design the schema system, **Then** the system uses plugin-based architecture where each treatment type has an isolated schema file (hemodialysis.schema.json, future: dialysis_full.schema.json).

2. **Given** new treatment type is added, **When** I create a new schema, **Then** the system loads the schema dynamically based on treatment type selection without code changes to core application logic.

3. **Given** schema versioning is needed, **When** I update a schema, **Then** the system includes schema version field (e.g., "version": "1.0.0") and supports backward compatibility with migration utilities.

4. **Given** schema is loaded, **When** the app initializes, **Then** the system validates schema structure, checks for required fields (fieldName, dataType, required, validationRules), and rejects invalid schemas with error message.

5. **Given** multiple schemas exist, **When** I manage schemas, **Then** the system stores schemas in /schemas directory with naming convention: {treatment_type}.schema.json and provides schema registry for lookup.

## Edge Cases
* What happens when schema file is missing? (Display error "Treatment type schema not found"; prevent treatment initiation; log missing schema error)
* How does the system handle schema conflicts? (Validate unique field names within schema; prevent duplicate field IDs; enforce schema validation rules)
