# User Story - US_036

## Story ID
* ID: US_036

## Story Title
* Hemodialysis Form Schema Definition

## Description
* As a developer, I want a JSON schema definition for the Hemodialysis form with field properties and validation rules, so that the form structure is well-defined and extensible for future treatment types.

## Acceptance Criteria
1. **Given** I need form structure, **When** I create the schema file, **Then** the system has a JSON schema with 15 fields: 9 Demographics & Admission fields (MRN, First Name, Middle Initial/Name, Last Name, DOB, Admission/Encounter Number, Gender, Treatment Location, Room Number) and 6 Clinical Intake fields (HBsAg, HBsAg Date Drawn, HBsAg Source, HBsAb Immune Value, HBsAb Date Drawn, HBsAb Source).

2. **Given** field properties are needed, **When** I define each field, **Then** the system specifies data type (String, Integer, Date, Enum, Boolean), required/optional flag, validation rules, and regex patterns for each field.

3. **Given** enum fields exist, **When** I define dropdown options, **Then** the system includes enum values: Gender (Male/Female), Treatment Location (OR/Bedside/ICU-CCU/ER/Multi-Tx Room), HBsAg (Positive/Negative/Unknown), HBsAb Source (Hospital/Davita Patient Portal/Non-Davita Source).

4. **Given** field ordering is needed, **When** I configure the schema, **Then** the system defines sequential field ordering (1-15) for conversation flow with default values for optional fields.

5. **Given** extensibility is required, **When** I design the schema architecture, **Then** the system uses modular plugin-based design with schema version field to support future treatment types and schema migrations.

## Edge Cases
* What happens when schema version changes? (Include schema version field; implement migration logic to transform old data to new schema)
* How does the system handle adding new fields in future? (Design additive schema changes; maintain backward compatibility; mark new fields as optional initially)
