# User Story - US_007

## Story ID
* ID: US_007

## Story Title
* Mock Data Models Definition

## Description
* As a developer, I want JSON-based mock data models for Patient, TreatmentSession, User, Hospital, AuditLog, and FormSchema, so that I can store and retrieve structured data in MMKV storage.

## Acceptance Criteria
1. **Given** I need patient data structure, **When** I define the Patient model, **Then** the system has a TypeScript interface with fields: mrn (string), firstName (string), middleName (string), lastName (string), dob (Date), gender (enum), admissionNumber (string), treatmentLocation (enum), roomNumber (string).

2. **Given** I need session tracking, **When** I define the TreatmentSession model, **Then** the system has fields: sessionId (string), patientMrn (string), formData (object), completionStatus (enum), lastFieldIndex (number), createdAt (Date), updatedAt (Date).

3. **Given** I need user authentication, **When** I define the User model, **Then** the system has hardcoded mock credentials with fields: userId (string), username (string), passwordHash (string), role (enum), hospitalId (string).

4. **Given** I need hospital context, **When** I define the Hospital model, **Then** the system has hardcoded mock hospitals with fields: hospitalId (string), name (string), code (string), address (string).

5. **Given** I need audit tracking, **When** I define the AuditLog model, **Then** the system has fields: logId (string), timestamp (Date), userId (string), actionType (enum), entityType (string), entityId (string), beforeValue (object), afterValue (object), ipAddress (string).

## Edge Cases
* What happens when model schema changes in future versions? (Implement schema version field; provide migration utility for data transformation)
* How does the system handle invalid JSON during deserialization? (Add JSON validation before parsing; log errors and return null with error flag)
