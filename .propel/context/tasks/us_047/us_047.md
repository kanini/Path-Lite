# User Story - US_047

## Story ID
* ID: US_047

## Story Title
* LLM Response Caching

## Description
* As a developer, I want LLM response caching for identical questions within a session, so that latency is reduced and Azure OpenAI API costs are minimized.

## Acceptance Criteria
1. **Given** a question is asked, **When** the same question is asked again in the same session, **Then** the system retrieves the cached LLM response instead of making a new API call, reducing latency to <100ms.

2. **Given** cache is implemented, **When** I store responses, **Then** the system uses in-memory cache with question text as key and LLM response as value, with session-scoped lifetime (cleared on session end).

3. **Given** cache lookup occurs, **When** I check for cached response, **Then** the system performs case-insensitive comparison, normalizes whitespace, and matches questions with >95% similarity.

4. **Given** cache hit occurs, **When** cached response is retrieved, **Then** the system logs cache hit event, returns cached structured output, and skips Azure OpenAI API call.

5. **Given** cache miss occurs, **When** no cached response exists, **Then** the system makes Azure OpenAI API call, stores response in cache, and uses cached value for subsequent identical questions.

## Edge Cases
* What happens when cache memory is full? (Implement LRU eviction; limit cache to 100 entries; clear oldest entries when limit reached)
* How does the system handle cache invalidation? (Clear cache on session end; provide manual cache clear option; invalidate cache on form schema changes)
